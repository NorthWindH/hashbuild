## Deviations from plan

One deviation, found via test failure during execution, not anticipated by the plan:

- **What the plan said:** `next_action_for_task` should call `_resolve(task["steps"], ref, on_exhausted="plan_task")` unconditionally (§2's literal code block).
- **What was done instead:** `on_exhausted` is now computed as `"plan_task" if not task["steps"] else "steps_complete"` before the `_resolve` call.
- **Why:** The plan's own §1 design decision explicitly states plan_task and steps_complete "share one implementation path... the only difference is *why* it's exhausted (zero steps ever existed vs. every step is done), which is threaded through as a parameter." But the literal top-level call in §2 hardcoded `on_exhausted="plan_task"` in all cases. Under that literal code, a task with one step that is executed and fully reviewed (closed) — i.e. AC 2.5's `steps_complete` case reached directly, not via the `review_or_next` branch's internal recursion — would fall through `_resolve`'s for-loop (no early return fires, since the step is fully done) and hit the `on_exhausted == "plan_task"` branch, wrongly reporting `stage="plan_task"` with an `/hb-task-plan` invocation for a task whose steps already exist and are finished. This was caught by `test_next_action_stage_steps_complete` failing with `AssertionError: assert 'plan_task' == 'steps_complete'`.
- **Why the result is still correct:** The fix makes the code match the plan's own stated intent (the "why it's exhausted" prose) rather than its literal-but-buggy pseudocode. `test_next_action_stage_plan_task` (zero steps) and `test_next_action_stage_steps_complete` (one step, fully done) both pass under the fix, and the `review_or_next` branch's internal recursive call — which already correctly hardcodes `on_exhausted="steps_complete"` for the "no next step" case — is unaffected since it's a separate call site.

No other deviations. All file/module boundaries, the CLI surface, the JSON/markdown rendering shapes, and the precedence chain's step-level logic match the plan exactly.

**Manual stage walk** (fresh `mktemp -d`, `hb-sdk` invoked directly):

```
=== next-action: no active tasks ===
- Start a new task with `/hb-task-create <author/task-id>`.
=== next-action: plan_task ===
- Run `/clear`, then `/hb-task-plan hasan/abc-1` to plan `hasan/abc-1` into steps.
=== next-action: plan_step ===
- Run `/clear`, then `/hb-task-step-plan hasan/abc-1/step-0` to plan the next step.
=== next-action: execute_step ===
- Run `/clear`, then `/hb-task-step-execute hasan/abc-1/step-0` to execute the plan.
=== next-action: review_or_next (flattened, no next step) ===
- Step `hasan/abc-1/step-0` is executed — review it or move on.
  - Review this step: `/hb-task-step-review-init hasan/abc-1/step-0`
  - Add more steps: `/hb-task-step-add hasan/abc-1`
  - Update the plan: `/hb-task-plan hasan/abc-1`
  - Archive the task: `/hb-task-archive hasan/abc-1`
=== next-action: steps_complete (json) ===
{
  "task": "hasan/abc-1",
  "stage": "steps_complete",
  "message": "All steps for `hasan/abc-1` are executed and reviewed — add more steps, update the plan, or archive.",
  "invocation": null,
  "choices": [
    {"label": "Add more steps", "invocation": "/hb-task-step-add hasan/abc-1"},
    {"label": "Update the plan", "invocation": "/hb-task-plan hasan/abc-1"},
    {"label": "Archive the task", "invocation": "/hb-task-archive hasan/abc-1"}
  ]
}
=== summarize --format md next action section (steps_complete state) ===
## Next Action

- All steps for `hasan/abc-1` are executed and reviewed — add more steps, update the plan, or archive.
  - Add more steps: `/hb-task-step-add hasan/abc-1`
  - Update the plan: `/hb-task-plan hasan/abc-1`
  - Archive the task: `/hb-task-archive hasan/abc-1`
```
