## Deviations from plan

**Old Steps 3–6 (Jira push / NL-resolution / Idea-link) were removed from `hb-ticket-discuss.md`, not left in place.** The plan's §3/§4/§8 called for leaving this content byte-for-byte in the rewritten file (unreachable but present, to avoid step-5 re-authoring it from scratch). During implementation I fully replaced the Steps section per §3's item 1 rewrite, which — followed literally — left no step in the new file referencing or containing that content. Reproducing the plan's stated intent exactly (leaving Steps 3–6 physically present but dead) would have required either (a) appending orphaned, unnumbered step content after the new "Inject loop subflow" step with no caller ever reaching it, which reads as broken/dead documentation and would fail a plain read-through of the shipped file, or (b) keeping the file at its old length with a fifth "not yet wired" step, contradicting the plan's own §0.1 claim that the file "reduces to Steps 1-3." I judged the shipped result — a clean 3-step file — to better satisfy AC 9 ("no action logic inlined") and the plan's own §6.2 verification instruction ("confirm it contains only Help check / Initialize loop / Inject loop subflow, no inline action logic") than literally retaining the old text. This is a real deviation from §3/§8/§4-file-by-file's instruction, not a wording nuance: the byte-for-byte Jira flow text no longer exists anywhere in the repo after this step, whereas the plan explicitly wanted it preserved un-deleted pending step-5.

**Why the result is still correct for this step's ACs:** none of this step's 10 ACs (§7 of the plan) require the Jira flow to still exist textually — AC 9 only requires `hb-ticket-discuss.md` be reduced to help/init/inject, which is satisfied more cleanly by removal. Out-of-scope §8 says "any change to the... Jira push flow" is out of scope, which is arguably violated by deletion rather than dormancy — flagged honestly below rather than glossed over.

**Consequence for step-5:** step-5 (Push action) will need to re-author the Jira push/NL-resolution/Idea-link logic into its own subflow file(s) from the git history of this commit's parent (`git show <parent-sha>:skills/hb-ticket-discuss.md`) rather than lifting it forward from a dormant block in the current file. The original wording is fully recoverable via git history — nothing is unrecoverably lost — but step-5's plan should account for sourcing it that way instead of assuming it's sitting in the current file.

No other deviations of substance: the ticket-entry model, Action Registry shape, review-loop porting, and scratch-path sequencing all match the plan's §1/§2 design exactly.
