## Deviations from plan

**`_parse_review_open` — exact match replaced with substring + header skip**

*Plan said:* `if status_val not in REVIEW_CLOSED_STATUSES: return True`

*What was done:* Added an explicit `continue` for header cell values (`"status"`, `"resolution"`), then used `any(kw in status_val for kw in REVIEW_CLOSED_STATUSES)` (substring match) instead of exact membership check.

*Trigger:* The existing review.md format in this repo uses a 2-column table `| ID | Resolution |` with resolution values like `"✅ Addressed — <desc>"`. Exact match on `"addressed"` would fail for these values, causing all actual review files to appear "review-open". Additionally, the standard Markdown header row `| ID | Status | Description |` contains "Status" in cols[2], which the plan's algorithm would treat as an open item before reaching any data row.

*Why the result is still correct:* The substring match correctly identifies `"✅ addressed — ..."` as closed (contains "addressed"), and `"open"` / other non-status text as open (no closed keyword present). The header skip explicitly guards both 2-column ("resolution") and 3-column ("status") formats. All 10 step-status tests pass, including case-insensitive, mixed, and no-row cases.

**Test setup — `task_step_add` idempotent behavior with manually created step folders**

*Plan said (implied):* create skeleton step-0 by calling `sp0.mkdir()`, then call `task_step_add` to get step-1 and step-2.

*What was done:* Used `task_step_add` for all steps, then deleted `ticket.md` from step-0 to achieve skeleton status.

*Trigger:* `task_step_add` is idempotent by ticket — when step-0 exists but has no `ticket.md`, it populates step-0 rather than creating step-1, so the second `task_step_add` would only produce step-1 (not step-2 as expected).

*Why the result is still correct:* The behavior tested is identical; the only change is how the skeleton state is reached (delete ticket vs. create bare directory). Test outcomes are the same.
