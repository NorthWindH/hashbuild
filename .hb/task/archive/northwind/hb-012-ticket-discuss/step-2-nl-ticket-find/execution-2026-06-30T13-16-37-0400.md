## Outcome

Before this step, `/hb-ticket-discuss` asked the user to pick "create new / update existing / no" and then resolved each Jira field in sequence through sequential prompts — meaning "update the auth refactor story in MOBILE" still forced the user through a menu and a bare `issueIdOrKey` prompt with no Jira search. After this step, the user describes the Jira target in plain English once; the agent resolves fields against the live Jira (project lookup, JQL search, or direct key retrieval), presents the full resolved set for confirmation, and loops on decline. The 3-way menu is gone.

Change is additive/refactor: new Step 4 (NL resolution + confirmation loop) inserted between detect/collect (Step 3) and push (renumbered Step 5); push call preserved verbatim; Steps 6/7 (renamed from 5/6) content identical. No automated harness (markdown-procedure repo) — structural checks are authoritative.

**Design as built:** Step 3 collects `$NL_DESC` (or routes to fallback on "no"). Step 4 is a loop: (A) parse intent/extract partial fields, (B) resolve `cloudId` via `getAccessibleAtlassianResources`, (C) resolve remaining fields by path — create path uses `getVisibleJiraProjects` + `getJiraProjectIssueTypesMetadata`; update path uses `getJiraIssue` for explicit keys or `searchJiraIssuesUsingJql` for NL-described issues. (D) present resolved set, (E) user accepts/refines/supplies exact values/aborts. Ambiguous matches → numbered list; unresolved fields → explicit prompt; query errors → surface verbatim + prompt. Step 5 receives pre-resolved `$JIRA_FIELDS` and calls the push tool without any resolution logic.

## Deviations from plan

**Step 5 push wording simplified.** The plan said to preserve current Step 4's push call "verbatim". Current Step 4 had redundant cloudId resolve logic (calling `getAccessibleAtlassianResources` again) and per-field sequential prompts that are now entirely in Step 4. Carrying those verbatim into Step 5 would have duplicated resolution that already happened. Instead Step 5 is written as resolution-free: it receives `$JIRA_FIELDS` from Step 4 and calls the push tool directly. The Rovo parenthetical examples (`createJiraIssue`, `editJiraIssue`) are preserved as required by STEP-1-REVIEW-4. The push tool call wording ("call the MCP's create-issue tool / edit-issue tool … Atlassian Rovo example: …") is structurally identical to the original — no regression risk.
