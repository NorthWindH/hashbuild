## Outcome

The motivating case — a user who wants to draft a hashbuild-shaped `ticket.md` to paste into Jira (or hand off elsewhere) **without** materializing any `.hb/` task or step folder — is now resolvable. Before: the only entry points to the interactive ticket flow (`hb-task-create`, `hb-task-step-add`) both end by calling `hb-sdk task ...`, which writes into `.hb/task/active/...`; standalone generation was impossible. After: the new `skills/hb-ticket-discuss.md` drives the shared interactive subflow against a scratch path (`/tmp`) and prints the result to stdout for copy-paste, touching nothing under `.hb/`.

Scope boundary that held: **purely additive** — exactly one new file, no edits to any existing file (the shared subflow and its two existing callers are untouched). No SDK call, no commit of a repo artifact, no Jira integration (deferred to step 1). No build/test gate exists for this markdown-procedure repo; conformance was checked statically.

Step 4 prompts the user and notes Jira push lands in step 1. `allowed-tools` copies the siblings' `Write`/`Read`/`Edit` grants for `//tmp/*` and `//private/tmp/*` and omits `Bash(${CLAUDE_SKILL_DIR}/scripts/hb-sdk *)` and `Bash(git *)`, enforcing the no-SDK/no-commit invariant at the permission layer.
