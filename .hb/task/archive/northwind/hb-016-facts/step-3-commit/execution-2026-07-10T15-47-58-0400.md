## Deviations from plan

One deviation, in a verification check only — not in the shipped change.

- **What the plan said:** §6 step 4 ("Propagation check") expected
  `grep -rln "committing.md" skills/*.md | wc -l` to return `11`, based on a
  count taken during planning (§0, "Blast radius").
- **What was found instead:** running the same command now returns `9`
  (`hb-init.md`, `hb-task-archive.md`, `hb-task-create.md`,
  `hb-task-step-add.md`, `hb-task-step-execute.md`, `hb-task-step-plan.md`,
  `hb-task-step-review-address.md`, `hb-task-step-review-init.md`,
  `hb-task-unarchive.md`). A recursive variant
  (`grep -rln "committing.md" skills --include="*.md"`) returns `10`, the
  extra hit being `skills/references/references-toc.md` — a references index,
  not a skill that itself performs a commit step.
- **Why:** the plan's count of 11 was taken during step-3's own planning
  pass and isn't reconcilable from the current repo state (no skill file
  referencing `committing.md` was removed by this step or by any of steps
  0–2); most likely it was a miscount at plan time rather than a regression
  introduced here, since no commit in this task's history removes a
  `committing.md` reference from any skill file.
- **Why the result is still correct:** the actual AC (§7 of the plan) only
  requires that the note land once in `committing.md` and read generically
  — both hold regardless of how many skills currently reference the file.
  The propagation mechanism itself (skills reference `committing.md` by
  pointer, not by copy) is unaffected by the exact count: whether 9, 10, or
  11 skills reference it today, all of them pick up the new bullet the next
  time they're read, with zero per-skill edits. The count mismatch is a
  stale planning artifact, not a defect in the shipped change.

No other deviations.
